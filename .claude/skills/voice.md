# Voice — Yasser's Writing Style

Use this skill when writing any user-facing copy: marketing pages, blog posts, documentation descriptions, error messages, UI text, or any prose that will be read by developers.

## Voice Characteristics

**Tone**: Casual, honest, self-aware. Not corporate. Use "you" and "your" freely. Admit limitations when they exist. Don't pretend things are more polished than they are.

**Sentence structure**: Short sentences. Direct. Say what something does, then stop. If a sentence has a semicolon, it's probably two sentences. Parenthetical asides are fine (like this one).

**Technical honesty**: Explain WHY, not just what. Acknowledge tradeoffs. Respect competing tools. If something isn't built yet, say so. Never overpromise.

**What to avoid**:
- Corporate marketing speak ("leverage", "empower", "seamless", "revolutionary")
- Buzzword chains ("AI-powered cloud-native serverless platform")
- Passive voice where active is clearer
- Hedging with unnecessary qualifiers ("helps enable you to potentially...")
- Generic SaaS filler ("take your X to the next level")
- Exclamation marks in headlines (one per page max, and only if genuine)

**What works**:
- "You don't configure any of it" > "Infrastructure is managed for you"
- "When something breaks, you see which function failed" > "Comprehensive error visibility across your stack"
- "Just push your code and it goes live" > "Streamlined CI/CD deployment pipeline"
- "If you self-host, workflows need a persistence layer" > "Self-hosted workflows require infrastructure provisioning"

## Examples from Yasser's actual writing

Good (natural):
- "After a few days of headbanging, we ended up with three new backends"
- "I'm still sharpening my branding skills"
- "Is it significantly different? Not really. But it's less documentation and verbosity"
- "So yeah, I guess I convinced myself to keep using pikku"
- "That was a painful discovery"

Bad (corporate):
- "Fabric provides a comprehensive managed infrastructure solution"
- "Leverage our enterprise-grade deployment pipeline"
- "Seamlessly integrate with your existing workflow"

## When to apply

Every time you write text that a human will read. This includes:
- Page headlines and descriptions
- Feature descriptions
- Error messages and UI labels  
- Blog post drafts
- Documentation introductions
- Pricing descriptions
- Changelog entries

## The test

Read it out loud. If it sounds like a developer explaining something to another developer over coffee, it's right. If it sounds like it came from a marketing team's slide deck, rewrite it.
